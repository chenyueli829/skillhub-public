# 每日简讯（Daily Brief）

## 与 Newsletter 的区别

| 维度 | 每日简讯（本 skill） | `newsletter-generation` |
|------|----------------------|-------------------------|
| 篇幅 | 每条 1 句，全文可极短 | 多段、分节、可长文 |
| 结构 | emoji + 一句 + @ + 可选标签 + 分隔线 | 标题、导语、分节、深度摘要 |
| 场景 | 群推送、备忘录、快扫 | 邮件周刊、博客式读物 |

若用户要「一篇能读几分钟的周刊/通讯」，用 newsletter skill；若要「一条条划过去的一屏简讯」，用本 skill。

## 输出格式（必须遵守）

每条新闻 **2 行**（正文 + 分隔线），条目之间不强制空行。正文行使用 **Markdown** 输出，以便在 Cursor、GitHub、Notion 等环境中**可点击跳转**。

```markdown
[分类emoji] [主体] [动作/产品]，[一句价值或影响]。[@来源简称](https://…)  [🎬](https://…) [🖼️](https://…) [🔗](https://…)
------------------
```

- **分类 emoji**：每条开头 1 个，表示主题桶（见下表）。
- **正文**：一句话，**约 25–45 个汉字**；结构建议：`谁 + 做了什么 + 对读者有什么用`。
- **@ 信源（必须可点击）**：写成 Markdown 链接，指向**可核对的权威页**（官方公告、新闻稿、机构页面）；展示文案仍用 `@` 前缀风格，例如 `[@Anthropic](https://www.anthropic.com/news/…)`（渲染后为可点击的 `@Anthropic`）。无单一原文时，链到该机构官方新闻列表或主站对应栏目亦可，但须在自检中确认链接有效。
- **尾部标签（可选，按需空格分隔）**：
  - `🎬` 有视频 / 演讲 / 演示时，**推荐**将 `🎬` 本身做成链接（`[🎬](视频或含嵌入的页面URL)`），无独立视频页则链到正文同一条官方稿。
  - `🖼️` 有配图 / 信息图 / 海报时，同理可用 `[🖼️](url)` 指向含图页面。
  - `🔗` **有原文或公告值得点时必须可点击**：写作 `[🔗](https://…)`，URL 与 `@` 链接可相同（同一篇公告）或 `🔗` 指向更细的章节/附件页。
- **分隔线**：固定为 `------------------`（20 个 `-`），紧跟在正文下一行；分隔线**不要**包进链接里。

**合并写法（可选）**：若希望末尾整段「`@来源 🔗`」一次点进同一篇稿，可使用 `[@Anthropic 🔗](https://…)` 单链接，仍满足「末尾可点击访问」。

### 分类 emoji 参考表（全稿内保持一致）

| Emoji | 适用主题 |
|-------|----------|
| 💳 | 金融、支付、银行、投资、财务产品 |
| 🦙 / 📎 | 文档/检索/知识库、RAG、解析类产品 |
| ⚙️ | 架构、工程、中间件、开源基础设施 |
| 📊 | 报告、榜单、数据、研究综述 |
| 🤖 | 模型厂商、智能体、自动化、机器人 |
| 📈 | 增长、排名、市场、应用数据 |
| 🔬 | 论文、科研、评测方法 |
| 🏛️ | 政策、合规、监管 |

同一期简讯里，**每个 emoji 桶不要混用含义**；若某类没有合适图标，可复用最接近的一格，但全稿要统一。

## 写作规则

1. **一条一事**：不合并两条独立新闻；重复主题选最新或最重要的一条。
2. **可核实**：数字、日期、「第几名」等必须有依据；不确定则弱化表述或暂不写。
3. **不标题党**：不用夸张结论；「有望」「或」等保留不确定性。
4. **语言**：默认 **中文**；产品名、API、会议名可保留英文。
5. **顺序**：默认 **重要度 / 时效** 靠前；用户指定领域时按指定主题聚类。
6. **条数**：默认 **6–12 条**；用户指定条数则遵从。

## 工作流程

1. **确认**：领域（如 AI / 泛科技）、语言（默认中文）、条数、日期（用于检索与刊头）。
2. **检索**：用当前日期做时间锚点，检索近 24–72 小时高可信来源（官方博客、权威媒体、学术/机构发布）。
3. **起草**：按「正文行 + 分隔线」逐条输出。
4. **落盘**：将**与用户展示内容一致**的完整 Markdown 写入本 skill 目录下的 `outputs/`（见「输出与保存」命名规则）；`outputs` 不存在则创建。
5. **自检**：见下方清单。

## 刊头（可选）

若用户需要「日期 + 刊名」，在第一条前增加：

```text
# [刊名，如：科技快讯] · YYYY-MM-DD
```

或一行：

```text
—— YYYY年MM月DD日 · 每日简讯 ——
```

## 质量清单

- [ ] 每条仅两行：第二行必为 `------------------`
- [ ] 每条仅一句正文，含一个分类 emoji 与 `@` 信源
- [ ] `@` 信源与（若出现）`🔗` 均为 Markdown 链接，在预览中可点击打开权威页
- [ ] 🎬🖼️🔗 仅在有对应材料时添加；若加了 `🎬`/`🖼️`/`🔗`，应对应有效 URL
- [ ] 无编造事实；存疑项已弱化或删除
- [ ] 全篇 emoji 分类口径一致
- [ ] 已保存至 `outputs/`，文件名符合 `YYYY-MM-DD-daily-brief-{话题}.md`

## 输出与保存

- **必须先展示、后落盘（或同时）**：向用户输出完整 Markdown 简讯后，**必须**将同一份内容保存到本 skill 下的 **`outputs/`** 目录。
- **命名规则（必须遵守）**：`YYYY-MM-DD-daily-brief-{话题}.md`  
  - **日期**：与刊头或检索锚点一致的公历日期，格式 `YYYY-MM-DD`。  
  - **话题**：简短主题 slug，建议小写英文与连字符（如 `ai`、`overseas-ai`、`frontier-models`），避免空格；若用户只用中文描述主题，可译为简短 slug 或在文件名中使用简短中文（优先 ASCII slug 以利跨平台）。  
  - 示例：`outputs/2026-04-17-daily-brief-ai.md`、`outputs/2026-04-16-daily-brief-overseas-ai.md`。
- 用户若要求英文简讯：保持同一两行结构，正文改为英文一句，`@` 信源与 `🔗` 等仍须为 Markdown 可点击链接，**同样保存到 `outputs/`** 并沿用上述文件名规则。

## 协作

- 需要深度背景或单题长文时，可另用 `deerflow-deep-research`；本 skill 只负责 **高密度短讯列表**。
