### 第二步：接收文案，整理确认表（必须输出表格）

收到文案后，**立即整理成确认表**发给用户，格式：

> 梳理如下，请确认：
>
> | 页码 | 页面类型 | 参考图 | 内容摘要 |
> |------|---------|--------|---------|
> | 封面 | 首页 | cover | 主标题 / 副标题 |
> | P1 | 内容页 | content | 标题 / 要点摘要 |
> | P2 | 内容页 | content | 标题 / 数据摘要 |
> | ... | ... | ... | ... |
> | 尾页 | 结尾页 | closing | 总结 / 致谢 |
>
> 共 X 页，请确认页面分配是否正确，确认后说"启动生成"！

**页面类型判断规则：**
- 首页/封面 → **cover**
- 有时间轴/流程/组织架构 → **content**（导航类）
- 有数据表格/数字对比 → **content**（图表类）
- 有场景故事/对话泡泡 → **content**
- 纯文字列表/要点 → **content**
- 结尾/致谢 → **closing**

> **说明：** 除白板板书风格外，其余风格统一使用三种参考图：cover / content / closing。白板板书风格的 content 页统一使用 content_new.jpg。

等用户确认后再进入第三步。

### 第四步：逐页生成并发预览

**每页命令模板（严格执行）：**

```bash
python {baseDir}/scripts/generate_image.py --prompt "{提示词前缀}{页面专属提示词}" --filename "{输出文件名}.jpg" --resolution 2K --aspect-ratio 16:9 -i "{参考图绝对路径}"
```

> Windows CMD 下写成一行；PowerShell 下换行用反引号（`` ` ``）。

**关键参数说明：**
| 参数 | 说明 |
|------|------|
| `-i` | 传入参考图（image-to-image 模式，**必须带**） |
| `--resolution` | 固定用 `2K`（支持 `2K` / `3K`） |
| `--aspect-ratio` | 固定用 `16:9` |
| `--filename` | 必须以 `.jpg` 结尾 |

每次调用只生成一页。

---

### 提示词来源

**所有风格的提示词均从 `{baseDir}/styles/styles.json` 中读取。**

每个风格在 `styles.json` 中有 `style_prompts` 配置，包含 `cover`、`content`、`closing` 等页面类型的专用提示词。

**使用方法：**
1. 根据用户选择的风格，从 `styles.json` 读取对应风格的 `style_prompts`
2. 根据页面类型（cover / content / closing）选取对应提示词
3. 将提示词作为 `--prompt` 的前缀，拼接页面文案内容

**白板板书风格额外说明：**
- 白板风格支持更细分的页面类型：`cover / text / content / chart / navigation / closing`
- 但所有内容类页面（text / content / chart / navigation）统一使用 `content_new.jpg` 作为参考图

### 参考图路径（按风格）

**白板板书风格** `{baseDir}/styles/whiteboard/`：
```
├── cover.jpg         ← 首页
├── content_new.jpg   ← 所有内容页统一使用
└── closing.jpg       ← 尾页
```

**使用 .png 参考图的风格：**
- 黑板报风格 `{baseDir}/styles/黑板报风格/` → cover.png / content.png / closing.png
- 旧画报风格 `{baseDir}/styles/旧画报风格/` → cover.png / content.png / closing.png

**其余 11 种风格** `{baseDir}/styles/{风格文件夹名}/`：
```
├── cover.jpg         ← 首页
├── content.jpg       ← 内容页
└── closing.jpg       ← 尾页
```

**风格与文件夹映射表：**

| 风格名 | 文件夹名 | 参考图格式 |
|-------|---------|----------|
| 白板板书 | `whiteboard` | jpg（content 用 content_new.jpg） |
| 光辉 | `光辉` | jpg |

| 画架 | `画架` | jpg |
| 立体 | `立体` | jpg |
| 黑板报风格 | `黑板报风格` | **png** |

| 毛毡风 | `毛毡风` | jpg |
| 拟物毛玻璃 | `拟物毛玻璃` | jpg |
| 医疗行业 | `医疗行业` | jpg |

| 湿壁画 | `湿壁画` | jpg |

**规则：**
- 每页必须带参考图（`-i`），不同页型严禁混用参考图
- 封面页必须用 cover 参考图，尾页必须用 closing 参考图
- 以脚本输出的 `MEDIA:` 行为准获取生成图片路径，不依赖 stderr 解析
- 只记录保存路径，不读取图片内容


**重要提示：**

当图像需要包含文本/字符时，请遵循以下规则以减少文本渲染错误：

1. **始终使用 `--resolution 3K`** — 更高的分辨率能显著提高文本准确性。上述所有命令示例均已默认使用 3K。
2. **在提示词中用双引号括起显示的文本** — 任何应出现在图像中的文本都必须在提示词字符串内用双引号括起来。例如，如果用户希望图像中出现文本 "Happy Birthday"，提示词应包含 `\"Happy Birthday\"`。

### 示例

```bash
python {baseDir}/scripts/generate_image.py --prompt "一张中间有文字 \"生日快乐\"，周围环绕着彩色气球的海报" --filename "poster.jpg" --resolution 3K
```

>注意：`\"生日快乐\"` 用转义双引号括起来了——这告诉模型将其作为图像中的文字文本呈现。

### 第五步：所有页确认后合成 .pptx

所有页面确认后，调用合成脚本：

```bash
python {baseDir}/scripts/build_pptx.py -i "slide1.jpg" "slide2.jpg" "slide3.jpg" -o "{workspace}/你的文件名.pptx"
```

> `build_pptx.py` 已内置跨平台支持，统一使用 `python` 调用。中文路径问题已在脚本内部处理。

**build_pptx.py 参数说明：**
| 参数 | 说明 |
|------|------|
| `-i img1.jpg img2.jpg ...` | 图片路径列表，**按页序排列** |
| `--dir ./某目录/` | 扫描目录下所有 jpg（按文件名字母序） |
| `-o output.pptx` | 输出路径，默认 `output.pptx` |
| `--width` / `--height` | 幻灯片尺寸（英寸），默认 13.33 x 7.5（16:9） |

**机器可读输出：** 脚本最后一行输出 `PPTX:/path/to/file.pptx`，可直接解析路径。

合成完成后将 .pptx 文件绝对路径告知用户。

## 常见错误（禁止）
- 不问风格直接问文案
- 收到文案不整理确认表直接生成
- 不传参考图（`-i`）或混用参考图
- 封面页使用通用前缀（如该风格有封面专用前缀）
- 省略提示词前缀
- 用错参考图（如内容页用了 cover.jpg）
- 忽略 styles.json 中的风格配置

## 页数限制与风险提示

### 建议页数
- **推荐范围**：10-20 页
- **单次上限**：20 页（含封面和尾页）

### 超过 20 页的风险
当 PPT 页数超过 20 页时，可能出现以下问题：
1. **生成中断**：长任务容易因网络波动或超时导致中断
2. **内存占用**：大量图片生成会占用较多系统资源
3. **预览困难**：一次性预览过多页面体验较差

### 处理建议
如果用户内容超过 20 页，在第二步确认表之后、进入第三步之前，主动提示：

>  当前共 **X 页**，超过 20 页建议分批次生成：
> - 方案 A：拆分为多个 PPT（如 Part 1 / Part 2），**需等前一批生成完成后再启动下一批**
> - 方案 B：精简内容，保留核心页面
>
> 请确认如何继续？
